## 1. R6.5 SSE 池化 + AI 缓存（最大头）

### R6.5.1 Redis 分布式锁 + 单 worker scheduler
**问题**：4 worker 模式下每个 worker 都启动 scheduler = 4 倍 cron 触发（一条提醒发 4 次）。

**修复**：用 Redis 锁（`SET NX EX 60`）让 1 个 worker 抢到锁跑 scheduler，其他 3 个跳过：
- 锁 key: `lock:scheduler:scheduler`
- TTL: 60s
- 持锁 worker 每 30s 用 Lua 续期（避免 TTL 过期被抢）
- 进程退出时 release（Lua 脚本：仅当 value == 自己的 token 才删）

**附加修**：`/cron/jobs` 端点不查本地 `_scheduler`（只在一个 worker 里有），改为查 Redis 锁 + 读 jobs 元数据（抢到锁的 worker 把 jobs 写到 `scheduler:jobs:list`）。

**实测**：
```
redis GET lock:scheduler:scheduler
  → 03a684fa11c68e2df455a61f9b6c3231 (TTL=46s)
GET /api/v1/cron/jobs
  → running: true, jobs: 3, lockOwner: "a526330a"
```

### R6.5.2 AI 端点结果缓存（5min TTL）
**问题**：AI mock 端点 /ai/ask/ask p95 = 2.9s，/ai/risk/scan p95 = 984ms。

**修复**：用 `@cache(key_prefix="ai:ask", ttl=300)` 装饰 `llm_ask` 和 `risk_scan`：
- key: `cache:ai:ask:<args_hash>`
- TTL: 300s
- 同一问题 5min 内不重算

## 7. 关键决策

1. **Redis 分布式锁而非单 worker 部署**——4 worker 模型不变，scheduler 用锁互斥，最小化架构变动
2. **Lua 脚本校验 token**——避免误删别人的锁（多 worker 持锁 token 各不同）
3. **cache 装饰器放 ai_client.py 内部**——业务层不感知，业务返回 dict 不带缓存副作用
4. **mock_get_user_by_state 按 state 末 4 位分配角色**——本地开发 + E2E 都能跑出不同角色
5. **企业微信 SSO mock 模式 fallback 到 admin**——避免 state → userid 找不到本系统用户时 401
6. **真接入等企业资质**——code 完整，等 corp_id/secret 配环境变量即可切真
