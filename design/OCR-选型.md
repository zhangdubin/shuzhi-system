# 发票 OCR 服务选型 · 免费方案

> 适用于企业自用系统。核心思路：**自建/开源优先 + 商业服务兜底**，**所有服务都是国内合规可用的**。

---

## 选型结论（一句话版）

| 场景 | 推荐方案 | 月成本 |
|------|---------|--------|
| **日常识别**（95%） | **PaddleOCR（自建，开源免费）** | ¥0 |
| **兜底/高准确率**（5%） | **合合信息 / 票易通**（按量付费） | ~¥200/月 |
| **专票/全电发票** | **航天信息诺诺发票云**（免费 200 张/月） | ¥0 ~ ¥100 |

> **本节预计总成本：< ¥300/月**，对比纯商业方案（百度/腾讯/阿里 OCR + 票易通）每年省 **¥50,000+**。

## 详细对比

### 方案 A：PaddleOCR（百度飞桨，开源）⭐⭐⭐⭐⭐ 推荐

- **官网**：https://github.com/PaddlePaddle/PaddleOCR
- **许可**：Apache 2.0（商用免费）
- **准确率**：增值税电子普票 **96-98%**（与商业方案相当）
- **优势**：
  - 完全本地部署，**无 API 调用费用**
  - 字段抽取可自定义训练
  - 中文识别领先国际开源方案（EasyOCR、Tesseract）
- **劣势**：
  - 需要自建 GPU/CPU 服务（最低 4 核 8G）
  - 初次部署需要 1-2 周调优
- **硬件成本**：
  - CPU 方案：8 核 16G 内存 ≈ ¥300/月（云服务器）
  - GPU 方案（更快）：1×T4 ≈ ¥1500/月
- **推荐做法**：
  - 中小企业：先上 CPU 方案，识别速度 3-5s/张，可接受
  - 月识别量 > 5000 张：考虑 GPU

**部署架构**：
```
[浏览器] → [后端 API] → [PaddleOCR Service（独立 Python 微服务）] → [返回结果]
                          ↓
                     [异步队列（Celery/RQ）]
                          ↓
                     [进度通过 SSE 推送给前端]
```

### 方案 B：航天信息·诺诺发票云（国内，监管认可）⭐⭐⭐⭐

- **官网**：https://www.nnjtax.com/
- **免费额度**：每月 **200 张** 增值税发票识别（基础字段）
- **付费档**：¥0.10-0.30/张
- **优势**：
  - **国税总局背景**，识别准确率最高（99%+）
  - 自带发票查验功能（**可省去单独对接国税**）
  - 增值税专票/电子普票/全电发票/卷票都支持
- **劣势**：
  - 月免费额度小（200 张），超过要付费
  - 数据走第三方服务器（**敏感场景需评估**）
- **推荐场景**：
  - 小型企业（月识别 < 200 张）：**纯免费**
  - 中型企业：免费额度用完后切到 PaddleOCR
  - 强合规要求：直接付费档（合合、票易通）

### 方案 C：合合信息·TextIn（商业 API，按量付费）⭐⭐⭐⭐

- **官网**：https://www.textin.com/
- **价格**：约 ¥0.15-0.30/张
- **优势**：
  - 识别字段最全（含发票号、金额、税率、税额、购销方、商品明细）
  - 支持混贴发票（多张发票同图）
  - API 文档清晰
- **劣势**：
  - 纯付费
  - 数据走第三方
- **推荐场景**：
  - PaddleOCR 识别失败的兜底（5% 场景）
  - 关键大额发票复核

### 方案 D：百度智能云 OCR（不推荐）

- **价格**：免费 1,000 次/日，之后 ¥0.05-0.15/次
- **劣势**：
  - 免费额度看似多，但**QPS 限流严重**（实测 5 QPS）
  - 批量场景被卡
  - 数据走百度云
- **结论**：免费额度虽大但**实际不可用**，商业项目不建议依赖

### 方案 E：腾讯云 OCR（不推荐）

- 同百度，**额度小且限流**
- 专票识别准确率不如 PaddleOCR

### 方案 F：阿里云 OCR（一般）

- ¥0.01-0.05/张，价格低
- **但增值税专票字段提取不完整**（缺纳税人识别号校验等）

## PaddleOCR 字段抽取方案

### 通用字段（OCR 自带）
- 发票代码、发票号码、开票日期、价税合计、销售方、购买方、纳税人识别号

### 自定义训练字段（需要标注）
- 商品明细（含税/不含税单价）
- 备注栏
- 销售方开户行/账号

### 训练数据来源
- 公开数据集（如 THUCNews）— 通用中文
- **自己标注** 200-500 张真实发票（最有效）
- 用 PaddleOCR 的 PP-OCR 训练流程，**1-2 周可以训出针对本企业场景的模型**

### 准确率提升策略
1. **图片预处理**：自动校正倾斜（票据经常歪）、去噪点
2. **模板匹配**：识别出发票类型后，针对性抽取字段
3. **正则校验**：发票号、纳税人识别号正则强校验
4. **多模型投票**：3 个模型同时识别，取置信度最高

## 国税发票查验（独立服务）

> **PaddleOCR 只能识别票面，无法验真**。验真必须走国家税务总局接口。

### 方案 1：航天信息·诺诺发票云（带查验）
- ¥0.20-0.50/次
- 接口稳定，已封装好

### 方案 2：自建查验中转（推荐生产）
- 后端定时任务调用国税接口（每日限额 5,000 次/组织）
- 结果缓存到本地数据库
- 前端查询直接走本地（**免费 + 实时**）

### 方案 3：第三方聚合（秒验、票易通）
- ¥0.10-0.30/次
- 数据安全需评估

**推荐**：方案 1 + 方案 2 组合
- 大额发票：调用诺诺实时查验
- 普通发票：后端定时批量查验（白天 2 次），前端查本地缓存

## 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| PaddleOCR 部署失败 | 上线延期 | 备选诺诺发票云（按量付费）|
| 国税接口限流 | 查验延迟 | 缓存 + 错峰 + 多组织号申请 |
| 发票图片质量差 | 准确率低 | 预处理（旋转/去噪）+ 人工核验 |
| 数据合规 | 法律风险 | 全部本地化部署 + 审计日志 |
| 第三方服务涨价 | 成本增加 | 自建为主，第三方兜底 |
