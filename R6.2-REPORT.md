## 2. R6.2.1 后端字段抽取器（postprocess.py）

**核心算法**：

```
PaddleOCR 原始行 (35 行) 
  ↓ _merge_lines（y_threshold=15 + X 范围重叠）
合并行 (10 行)  
  ↓ 正则匹配 + 启发式
8 字段 dict
```

**字段抽取策略**：
| 字段 | 策略 |
|---|---|
| invoiceNo | 8-20 位数字，最高 confidence |
| issueDate | `YYYY年MM月DD日` 或 `YYYY-MM-DD` 正则 |
| buyerName / sellerName | "名称：xxx" 模式 + 按 X 排序（X小=购方，X大=销方）|
| totalAmount | 包含"价税"或"小写"的最大金额 |
| taxAmount | 包含"税额" |
| amount | 包含"金额"且不含"税" |
| taxRate | `X%` 模式（0 < x ≤ 0.2）|

**修过的 3 个 bug**：
1. **多行合并太激进**：`y_threshold=25` 把左右两侧不同列合并成一行 → 改为 y_threshold=15 + X 范围重叠
2. **购销方位置颠倒**：原算法用"标签 + 下面公司名"——错（电子发票购方在**左**、销方在**右**，公司名在标签**右**边）→ 改为 "名称：xxx" 模式 + 按 X 排序
3. **金额被明细行污染**：OCR 把明细行"数量/单价/金额/税额"挤成短行，每行无 ¥ 符号，但 `RE_AMOUNT` 会匹配数字 → 加 ¥/￥ 前置过滤

## 3. R6.2.2 PaddleOCR 微服务

**架构**：
- 基础镜像：`python:3.10-slim`（清华源，避开 fastly 限流）
- 引擎：`paddlepaddle==2.6.2` + `paddleocr==2.6.1.3`
- 模型：`ch_PP-OCRv3`（CPU 轻量，3 个模型共 ~18MB）
- 框架：FastAPI + uvicorn（2 workers，但 PaddleOCR 单例需注意）

**端点**：
| 端点 | 用途 |
|---|---|
| `GET /health` | 健康检查（Docker healthcheck 用）|
| `GET /version` | 模型版本 |
| `POST /recognize` | 真实识别 + 字段抽取 |
| `GET /debug/recognize` | 调试用：返回 OCR 原始行（带位置）|

**BACKEND_URL 环境变量**：
- OCR 容器内 `localhost` 不通，需要把 `http://localhost:8000/static/...` 替换成 `http://shuzhi-backend:8000/static/...`
- 通过 `BACKEND_URL` 环境变量传入，正则替换

## 9. 关键决策

1. **PaddleOCR 2.6.1.3（不是 2.7.0.3）**——2.7 + paddlepaddle 2.6.2 protobuf 不兼容（"Cannot parse tensor desc"）
2. **PP-OCRv3（不是 v4）**——2.6.1.3 默认是 v3，v4 模型 protobuf 又是不一样的问题
3. **去 enable_mkldnn**——2.6.2 + mkldnn 报 "set_mkldnn_cache_capacity 不存在"
4. **y_threshold=15 + X 重叠**——避免左右两列文字被合并
5. **¥/￥ 前置过滤**——明细行无 ¥ 符号，靠这个过滤
6. **BACKEND_URL 环境变量**——OCR 容器内 localhost 不通，用容器名访问
7. **2GB 内存限制**——PP-OCRv3 模型 1.5GB + paddlepaddle 600MB = 2.1GB（实测 1.876GB 留 100MB buffer）
8. **start_period=90s**——首次启动要下载模型 + 加载，healthcheck 不能太快
